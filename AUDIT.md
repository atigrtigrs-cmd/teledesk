# TeleDesk — Комплексный аудит: проблемы и улучшения

**Дата:** 15 марта 2026  
**Версия:** 1.0  
**Автор:** Manus AI

---

## 1. Обзор системы

TeleDesk — CRM-система для управления коммуникациями с партнёрами через Telegram. Система включает интеграцию с Telegram MTProto (GramJS), Bot API, Bitrix24 CRM, AI-аналитику (LLM), SSE-нотификации и рассылку через бота.

| Метрика | Значение |
|---------|----------|
| Пользователей | 6 |
| Telegram-аккаунтов | 3 |
| Контактов | 1 468 |
| Диалогов | 1 590 |
| Сообщений | 337 382 |
| Серверный код | ~4 700 строк (routers + telegram + worker) |
| Клиентский код | ~9 300 строк (страницы) |
| Тестов | 27 (6 файлов) |

## 2. Критические проблемы

### 2.1. Отсутствие индексов на `contacts.telegramId`

Таблица `contacts` (1 468 строк) не имеет индекса на колонке `telegramId`, хотя это основной ключ поиска при каждом входящем сообщении. Каждое новое сообщение вызывает `SELECT ... WHERE telegramId = ?` — без индекса это full table scan. При росте базы до 10 000+ контактов это станет узким местом.

**Рекомендация:** Добавить уникальный индекс на `contacts.telegramId`. Также добавить составной индекс `(dialogId, createdAt)` на таблицу `messages` (337K строк) — сейчас каждый запрос сообщений для диалога использует только FK-индекс на `dialogId`, а сортировка по `createdAt` происходит в памяти.

```sql
CREATE INDEX idx_contacts_telegramId ON contacts(telegramId);
CREATE INDEX idx_messages_dialog_created ON messages(dialogId, createdAt);
```

### 2.2. AUTH_KEY_DUPLICATED — конфликт Telegram-сессий

Все три Telegram-аккаунта имеют ошибку `AUTH_KEY_DUPLICATED`. Это происходит когда одна и та же MTProto-сессия используется из двух процессов одновременно. Несмотря на реализованный `processLock` (distributed lock через БД), проблема сохраняется — вероятно из-за того, что Render при zero-downtime deploy запускает новый инстанс до остановки старого.

**Текущие меры:** 120-секундная задержка старта worker'а, process lock с heartbeat, cooldown после AUTH_KEY_DUPLICATED.

**Рекомендация:** Рассмотреть переход на отдельный background worker (сейчас suspended на Render) с выделенным IP, либо использовать Redis-based lock вместо MySQL для более надёжной координации. Альтернативно — при деплое сначала останавливать старый инстанс (rolling restart вместо blue-green).

### 2.3. Аватарки контактов — 99.6% без аватарок

Из 1 468 контактов только 6 имеют аватарки. Причины:

- **Bot API** может скачивать фото только тех пользователей, которые взаимодействовали с ботом (0% видимость в случайной выборке).
- **MTProto** (через GramJS) может скачивать фото любого контакта, но требует активных Telegram-клиентов, которые сейчас не подключаются из-за AUTH_KEY_DUPLICATED.
- Worker на Render не вызывает `bulkUpdateAvatars` автоматически после синхронизации.

**Что сделано:** Добавлен Bot API fallback во все точки скачивания аватарок, добавлена кнопка "Аватарки" в UI. Но основное решение — починить MTProto-подключение.

**Рекомендация:** После устранения AUTH_KEY_DUPLICATED — запустить `bulkUpdateAvatars` один раз для массового скачивания. Добавить автоматический вызов после завершения синхронизации в worker.ts.

## 3. Архитектурные проблемы

### 3.1. Монолитный `routers.ts` — 1 797 строк

Файл `server/routers.ts` содержит все tRPC-процедуры в одном файле. При текущем размере это затрудняет навигацию и увеличивает риск конфликтов при совместной разработке.

**Рекомендация:** Разбить на модули:
- `server/routers/accounts.ts` — управление Telegram-аккаунтами
- `server/routers/dialogs.ts` — диалоги и сообщения
- `server/routers/analytics.ts` — аналитика и статистика
- `server/routers/bot.ts` — LeadCash Bot и рассылка
- `server/routers/bitrix.ts` — Bitrix24 интеграция
- `server/routers/settings.ts` — настройки, теги, quick replies

### 3.2. Монолитный `telegram.ts` — 1 576 строк

Аналогичная проблема. Файл содержит MTProto-подключение, обработку сообщений, синхронизацию истории, скачивание аватарок и отправку сообщений.

**Рекомендация:** Разделить на:
- `server/telegram/client.ts` — управление клиентами, подключение
- `server/telegram/sync.ts` — синхронизация истории
- `server/telegram/messages.ts` — обработка входящих/исходящих
- `server/telegram/avatars.ts` — скачивание и хранение аватарок

### 3.3. Отсутствие пагинации сообщений

Процедура `dialogs.get` загружает **все сообщения** диалога одним запросом. При 337K сообщений в базе, некоторые диалоги могут содержать тысячи сообщений, что приводит к:
- Медленной загрузке
- Высокому потреблению памяти на клиенте
- Большому объёму трафика

**Рекомендация:** Реализовать cursor-based пагинацию (загрузка по 50 сообщений с "Загрузить ещё" при скролле вверх).

### 3.4. Автоответы не реализованы в runtime

Таблица `auto_replies` существует, CRUD-процедуры работают, UI для создания автоответов есть. Но **автоответы не срабатывают** при получении сообщений — в `telegram.ts` и `worker.ts` нет кода, который проверяет правила автоответов при входящем сообщении.

**Рекомендация:** В обработчике входящих сообщений (`handleNewMessage` в telegram.ts) добавить проверку:
1. `first_message` — если это первое сообщение от контакта
2. `outside_hours` — если сообщение пришло вне рабочих часов
3. `keyword` — если текст содержит ключевое слово

### 3.5. Рабочие часы не настроены

Таблица `working_hours` пуста (0 записей). Без настроенных рабочих часов автоответ `outside_hours` не может работать, а аналитика по времени ответа не учитывает нерабочее время.

## 4. Проблемы производительности

### 4.1. `dialogs.list` загружает до 500 диалогов

Процедура `dialogs.list` имеет `LIMIT 500`, но для каждого диалога делает JOIN с контактами и последним сообщением. При 1 590 диалогах это может быть медленным.

**Рекомендация:** Добавить пагинацию (по 50 диалогов), виртуальный скролл на клиенте, и кэширование `lastMessage` прямо в таблице `dialogs` (денормализация).

### 4.2. Отсутствие мемоизации в Messages.tsx

Компонент `Messages.tsx` (1 209 строк) — самый большой и самый часто используемый. При этом используется только один `useCallback` и ноль `useMemo`. Каждый ре-рендер пересоздаёт все вложенные компоненты.

**Рекомендация:** Извлечь `DialogList`, `ChatView`, `ContactPanel` в отдельные компоненты с `React.memo`. Мемоизировать тяжёлые вычисления (фильтрация диалогов, форматирование дат).

### 4.3. SSE-подключение загружает все диалоги для подсчёта непрочитанных

В `AppLayout.tsx` есть запрос `trpc.dialogs.list.useQuery({ status: "all" })` только для подсчёта `unreadCount`. Это загружает все 500 диалогов с контактами и сообщениями только чтобы посчитать badge.

**Рекомендация:** Создать отдельную лёгкую процедуру `dialogs.unreadCount` которая возвращает только число.

## 5. Проблемы безопасности

### 5.1. Нет rate limiting на tRPC-процедурах

Процедуры `auth.register`, `auth.login`, `accounts.startPhoneLogin` не имеют rate limiting. Единственное ограничение — Telegram FLOOD_WAIT при отправке кодов.

**Рекомендация:** Добавить middleware для rate limiting на auth-процедуры (например, 5 попыток в минуту по IP).

### 5.2. Bitrix24 webhook URL хранится в открытом виде

Webhook URL для Bitrix24 содержит секретный токен и хранится в таблице `bitrix_settings` без шифрования. Любой пользователь с доступом к БД может получить полный доступ к Bitrix24 CRM.

**Рекомендация:** Хранить webhook URL зашифрованным или через environment variable.

### 5.3. Telegram session strings в БД

Session strings для MTProto хранятся в `telegram_accounts.sessionString`. Это эквивалент полного доступа к Telegram-аккаунту.

**Рекомендация:** Шифровать session strings at rest (AES-256 с ключом из env).

## 6. UX/UI проблемы

### 6.1. Нет мобильной адаптации

Страница `Messages.tsx` использует фиксированную 3-колоночную раскладку без responsive breakpoints. На мобильных устройствах интерфейс непригоден.

**Рекомендация:** Реализовать mobile-first layout: на мобильных — список диалогов → чат → панель контакта (как в Telegram). Использовать `useMobile()` хук для переключения.

### 6.2. Страница "Воронки" — заглушка

`FunnelsPage.tsx` показывает только "Скоро". Это создаёт впечатление незавершённого продукта.

**Рекомендация:** Либо реализовать Kanban-доску для управления статусами диалогов, либо убрать из навигации до готовности.

### 6.3. Мёртвый код — неиспользуемые страницы

Файлы `Analytics.tsx`, `ComponentShowcase.tsx`, `Dashboard.tsx`, `DialogDetail.tsx`, `Inbox.tsx`, `Settings.tsx` не подключены в `App.tsx` (заменены на `AnalyticsPage.tsx`, `SettingsPage.tsx`, `Messages.tsx`).

**Рекомендация:** Удалить неиспользуемые файлы для чистоты кодовой базы.

### 6.4. Нет empty state для аналитики

При отсутствии данных за выбранный период аналитика показывает нули без объяснения. Пользователь не понимает — нет данных или ошибка.

**Рекомендация:** Добавить информативные empty states с подсказками ("Нет данных за выбранный период. Попробуйте выбрать другой диапазон").

## 8. Инфраструктура

### 8.1. Render free tier — cold starts

Render free tier усыпляет сервис после 15 минут неактивности. Первый запрос после сна занимает 30-60 секунд. Telegram-клиенты при этом теряют соединение.

**Рекомендация:** Перейти на paid tier ($7/мес) или добавить health check ping каждые 10 минут.

### 8.2. Нет мониторинга и алертов

Нет системы мониторинга для отслеживания:
- Статуса Telegram-подключений
- Количества необработанных сообщений
- Ошибок синхронизации
- Времени ответа API

**Рекомендация:** Добавить `/api/health` endpoint с метриками, настроить UptimeRobot или аналог.

### 8.3. Нет бэкапов БД

Данные (337K сообщений, 1.5K контактов) не бэкапятся автоматически.

**Рекомендация:** Настроить автоматический mysqldump в S3 (ежедневно).

## 10. Заключение

TeleDesk — функциональная CRM с хорошей базовой архитектурой (tRPC, Drizzle ORM, SSE). Основные проблемы связаны с масштабированием (отсутствие индексов и пагинации), надёжностью Telegram-подключений (AUTH_KEY_DUPLICATED) и незавершёнными фичами (автоответы, воронки). Приоритетные улучшения — починка MTProto-сессий, добавление DB-индексов и пагинация — дадут наибольший эффект при минимальных затратах.
