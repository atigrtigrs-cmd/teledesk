---

## 1. Обзор

TeleDesk — CRM-система для управления Telegram-коммуникациями в CPA-партнёрской сети LeadCash. Проект включает подключение Telegram-аккаунтов через MTProto, синхронизацию диалогов, отправку сообщений, интеграцию с Битрикс24, AI-аналитику и бот-рассылки. На момент аудита в базе данных содержится 1 468 контактов, 1 590 диалогов и 337 382 сообщения.

Аудит проведён с четырёх перспектив: **разработчик** (архитектура, качество кода), **тестировщик** (покрытие, edge cases), **безопасник** (уязвимости, утечки данных) и **дизайнер** (UX, доступность, адаптивность).

## 3. Детальный разбор критических проблем

### 3.1 Render Free Plan и MTProto

Сервис развёрнут на Render с тарифом `free` (файл `render.yaml`). Бесплатный план Render автоматически останавливает сервис после 15 минут без входящих HTTP-запросов. При остановке процесса все MTProto TCP-соединения с серверами Telegram обрываются, worker-интервалы (polling каждые 30с, keepalive каждые 3 мин, watchdog каждые 5 мин) прекращают работу. После пробуждения сервис должен заново установить все соединения, что занимает 15-30 секунд и может привести к потере сообщений, пришедших за время сна.

**Рекомендация:** Перейти на платный план Render Starter ($7/мес) который обеспечивает always-on работу. Альтернатива — внешний cron-сервис (UptimeRobot, cron-job.org) который пингует `/api/health` каждые 10 минут, но это ненадёжно и не гарантирует стабильность MTProto.

### 3.2 Отсутствие пагинации сообщений

Процедура `messages.list` выполняет запрос `SELECT * FROM messages WHERE dialogId = ? ORDER BY createdAt` без `LIMIT`. При 337 382 сообщениях в базе, диалог с 1000+ сообщениями возвращает все строки за один запрос. Это создаёт нагрузку на БД, сеть и клиентский рендеринг. Аналогичная проблема существует в `dialogs.list` — все 1 590 диалогов загружаются одним запросом.

**Рекомендация:** Реализовать cursor-based пагинацию: загружать последние 50 сообщений, при скролле вверх подгружать следующую порцию. Для диалогов — виртуализированный список с подгрузкой по 30 штук.

### 3.3 Публичный debugStatus

Процедура `debugStatus` (строка 150 в `routers.ts`) объявлена как `publicProcedure`, что означает доступ без аутентификации. Она возвращает: ID и юзернеймы Telegram-аккаунтов, их статусы и ошибки, значение `NODE_ENV`, время работы сервера (`uptime`). Эта информация может быть использована для разведки перед атакой.

**Рекомендация:** Изменить на `protectedProcedure` и добавить проверку `ctx.user.role === 'admin'`.

### 3.4 Утечка Bitrix Webhook URL

Процедура `bitrix.get` возвращает полный объект настроек Битрикс24, включая `webhookUrl`. Webhook URL Битрикс24 — это фактически API-ключ с полным доступом к CRM: чтение/запись сделок, контактов, компаний. Любой авторизованный пользователь системы получает этот URL через DevTools.

**Рекомендация:** Исключить `webhookUrl` из ответа `bitrix.get` или маскировать его (показывать только `****...последние4символа`). Использовать webhook только на серверной стороне.

### 3.5 Отсутствие ролевой модели

Все процедуры используют `protectedProcedure` без проверки роли. Это означает, что любой залогиненный пользователь (даже с ролью `user`) может: удалять Telegram-аккаунты, изменять настройки Битрикс24, управлять ботом и рассылками, видеть все диалоги всех аккаунтов. Поле `role` в таблице `users` существует (`admin` | `user`), но нигде не проверяется.

**Рекомендация:** Создать `adminProcedure` middleware и применить его ко всем административным операциям (управление аккаунтами, настройки, бот, Битрикс).

## 4. Аудит безопасности

| Вектор атаки | Статус | Детали |
|---|---|---|
| SQL-инъекция | Защищён | Drizzle ORM параметризует все запросы, raw SQL использует template literals с автоэскейпом |
| XSS | Защищён | React экранирует по умолчанию, `dangerouslySetInnerHTML` не используется |
| CSRF | Частично | Cookie-based auth с SameSite, но нет CSRF-токенов для мутаций |

| Input validation | Частично | Zod-схемы есть, но без ограничений длины строк |
| Session management | OK | JWT в httpOnly cookie, SIGTERM disconnect добавлен |

## 5. Аудит UX/Дизайна

**Мобильная адаптация** является наиболее серьёзной проблемой с точки зрения UX. Боковая панель навигации имеет фиксированную ширину 60px без возможности скрытия на мобильных устройствах. Страница сообщений (основной рабочий экран) использует трёхколоночную раскладку без адаптации — на экранах менее 768px интерфейс становится непригодным для использования.

**Пустые состояния** отсутствуют на 10 из 20 страниц. Когда пользователь впервые открывает разделы "Контакты", "Теги", "Быстрые ответы" или "Автоответы" — он видит пустой белый (тёмный) экран без объяснения что делать. Хорошая практика — показывать иллюстрацию с текстом "Нет данных" и кнопкой действия ("Создать первый тег").

**God-компонент Messages.tsx** (1 209 строк) содержит 5 логически независимых компонентов в одном файле: список диалогов, чат, панель контакта, аватар и хук ресайза. Это затрудняет навигацию по коду, замедляет HMR и делает невозможным lazy-loading отдельных частей.

**Дублирование страниц** создаёт путаницу: `Settings.tsx` (413 строк) и `SettingsPage.tsx` (289 строк) — две разные реализации настроек. Аналогично `Analytics.tsx` (711 строк) и `AnalyticsPage.tsx` (566 строк). Только одна из каждой пары подключена к роутеру.

## 6. Аудит инфраструктуры

**Worker** запускается in-process с основным сервером (не как отдельный процесс). Это правильное решение для предотвращения AUTH_KEY_DUPLICATED, но создаёт проблему: если worker зависнет на тяжёлой операции (синхронизация 1000+ диалогов), это может заблокировать event loop основного сервера и замедлить API-ответы.

**ProcessLock** (222 строки) реализует распределённую блокировку через БД с TTL 2 минуты. Механизм работает, но при Render free plan (где сервис может быть остановлен в любой момент) блокировка может "зависнуть" на 2 минуты после аварийной остановки, задерживая старт нового инстанса.

**SSE-подключения** не имеют лимита — `const clients = new Set<Response>()` растёт неограниченно. При 100+ одновременных вкладках или автоматизированной атаке сервер может исчерпать память. Рекомендуется ограничить до 50-100 подключений и 1-2 на пользователя.
